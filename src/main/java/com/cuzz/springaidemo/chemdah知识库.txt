修改或获取玩家的变量。
玩家变量帮助详见 Chemdah 文档。

在获取语法后使用 default 语句可在数据不存在时返回默认值，在追加（add）语句后使用 default 语句可在数据不存在时以该数据为基础进行追加（add）行为。

profile data {action} [(add|to) {action}] [default {action}]
profile data keys

profile data key
profile data key default 10
profile data key to 10
profile data key add 1
profile data key add 1 default 10
profile data keys
由 Chemdah 提供的公有动作
Profile Data 的剖析
通过上述语句示例，从中可以了解到 profile data  是用来对玩家变量进行增删改查的。
语句格式为：profile data {action} [(add|to) {action}] [default {action}]
其中 profile data 作为语句标识符，而 {action} 指代的是动作部分，意味着此处可以填入Kether语句。
而此处会将填入的Kether语句视作独立的语句去解析，当获取到这部分的返回值时，再传回给语句标识符使用。
此处不再赘述 {action} 部分的知识内容，详情请查看 Kether 烹饪食用指南。

对变量进行查询的方法是极其方便且直接的：
profile data key

此Kether语句会将名为 key 的变量数据直接返回给玩家。
后续的 [(add|to) {action}] 部分则用于对玩家变量数据进行操作；
例如语句示例中展示的两条语句：
profile data key to 10
profile data key add 1

此处将上述两条示例记为语句Ⅰ与语句Ⅱ。
语句Ⅰ的作用，是对名为 key 的变量数据进行操作，将其值变更为 10。
语句Ⅱ的作用，是对名为 key 的变量数据进行操作，使其值自增 1 。
此处名为 key 的变量数据，在合法的条件下，可以是任意值，如：
profile data 临时数据 to 10
profile data 持久数据 add 1

然而 default 部分也是常用的一环，在Kether动作列表中，已为读者列举出其作用。

”在获取语法后使用 default 语句可在数据不存在时返回默认值，在追加（add）语句后使用 default 语句可在数据不存在时以该数据为基础进行追加（add）行为。“

值得一提的是，default 部分也可以传入来自Kether语句的返回值。
Profile Data 的应用场景
profile data作为Chemdah自带的变量操作语句，在对话、任务两大部分是极为方便且实用的。
其不仅可作为用于判断的条件，同时也可以作为数据以控制玩家的对话及任务流程。
__option__:
  theme: 'chat'
  title: '测试对话'

global-flags:
  - LOOK_PLATER
  - NO_MOVE

测试对话-0:
  npc id: 'adyeshach 测试对话'
  when:
    - if: check profile data *测试对话 == null
      open: 测试对话-1
    - if: check profile data *测试对话 != null
      open: 测试对话-2

上述是用profile data语句进行判断的一个实例，用于控制玩家该与Adyeshach插件提供的NPC的进行哪一个对话。
重要提示：当使用profile data进行对话判断时，请在恰当的位置使用profile data语句对变量数据进行修改，以保证判断不会重复进行或导致流程重复。
profile data 也常适用于任务代理中，尤其是任务完成时的代理动作。
# 该任务可由:
# 指令"/chemdah data set %player_name% 数据名 数据值"更改以完成
# Kether语句"profile data 数据名 to 数据值"更改以完成
玩家数据:
  meta:
    name: '玩家数据'
    type: 'Chemdah'
  task:
    '0':
      objective: 'player data'
      goal:
        # kether条件代理
        # $: check profile data test > 5

        # 数据名
        key: 'test'

        # 数据值
        value: 3

        # 完成次数
        amount: 1
  agent:
    accepted: |
      title color *"&7- &8[&aWIKI示例&8] &7-" subtitle color *"&6玩家数据 - 已接受" by 10 20 10
    completed: |
			profile data test to 4
      title color *"&7- &8[&aWIKI示例&8] &7-" subtitle color *"&6玩家数据 - 已完成" by 10 20 10

在部分任务(如 player inventory)及部分语句(inventory check)中常需要对物品进行确切识别，因此了解如何判断物品显得十分重要。
多个判断可使用 "," 分割，如
/chs invoke inventory check "minecraft:diamond[name=abc,lore=123]" amount 1
请不要在判断中使用特殊符号，这可能会对判断结果产生影响。
材质判断
材质判断，识别的是物品的 material 值。
/chs invoke inventory check "minecraft:diamond" amount 1
在游戏中输入该指令，即可判断背包中 钻石 的数量是否大于等于1。
绿色部分便是物品识别的表达式内容。
而 minecraft:diamond 可看作 命名空间:材质名称，以便于理解。
更多材质名称请在游戏中使用按键 F3+H 查看。
名称判断
名称判断识别的是物品的展示名称。
/chs invoke inventory check "minecraft:diamond[name=钻石一号]" amount 1
在游戏中输入该指令，即可判断背包中 名称为 "钻石一号" 的钻石 的数量是否大于等于1。
[name=XX]，其中XX为需要判断的物品名称。
Lore判断
Lore判断识别的是物品的Lore描述。
/chs invoke inventory check "minecraft:diamond[lore=钻石一号]" amount 1
在游戏中输入该指令，即可判断背包中 Lore含有 "钻石一号" 的钻石 的数量是否大于等于1。
[lore=XX]，其中XX为需要判断的物品Lore描述。
耐久值判断
耐久值判断识别的是物品的耐久度。
/chs invoke inventory check "minecraft:diamond_sword[damage=6]" amount 1
在游戏中输入该指令，即可判断背包中包含 已损失的耐久度为6的钻石剑 的数量是否大于等于1。
[damage=6]，其中6为需要判断的物品已损失的耐久度。
别名 durability。
自定义模型数据值判断
自定义模型数据值判断识别的是物品的CustomModelData值。
/chs invoke inventory check "minecraft:diamond[custom-model-data=10000]" amount 1
在游戏中输入该指令，即可判断背包中包含 CustomModelData值为10000的钻石 的数量是否大于等于1。[custom-model-data=XX]，其中XX为需要判断的物品CustomModelData值。
附魔判断
附魔判断识别的是物品的附魔。
/chs invoke inventory check "minecraft:diamond_sword[enchant=DAMAGE_ALL]" amount 1
在游戏中输入该指令，即可判断背包中包含 已附魔锋利Ⅰ的钻石剑 的数量是否大于等于1。
[enchant=XX]，其中XX为需要判断的物品的附魔，此处附魔需要大写。
更多附魔可点击查阅。
别名 ench，enchants，enchantment。
药水判断
药水判断识别的是物品的药水效果。
/chs invoke inventory check "minecraft:potion[potion=SPEED]" amount 1
在游戏中输入该指令，即可判断背包中包含 速度的药水 的数量是否大于等于1。
[potion=XX]，其中XX为需要判断的物品的药水效果，此处药水名称需要大写。
兼容Bukkit中的customEffects。
更多药水效果可点击查阅。
别名 potions。

global-flags 全局标签
虽然插件自带的对话示例文件未包含该部分，但在插件源码路径(ink.ptms.chemdah.core.conversation.ConversationManager)
及 官方对话编辑器 内都有提及该部分。
[请注意，官方对话编辑器 中将 LOOK_PLAYER 标签 错编为 LOOK_PLATER，使用时请注意更改]
接下来将陈列所有全局标签及其作用。
global-flags 全局标签 格式
使用时请根据下列介绍取消对应注释以使用
__option__:
  theme: 'chat'
  title: 'Mr.{name}'
  global-flags:
    # - 'FORCE_LOOK'
    # - 'FORCE_DISPLAY'
    # - 'LOOK_PLAYER'
    # - 'NO_EFFECT'
    # - 'NO_EFFECT:SLOW'
    # - 'NO_EFFECT:BLINDNESS'
    # - 'NO_EFFECT:SOUND'
    # - 'NO_EFFECT:PARTICLE'
    # - 'NO_MOVE'
    # - 'NO_SKIP'
FORCE_LOOK 强制目视
该标签用于在触发对话时，强制让玩家目视对话目标NPC。
此效果也可由指令触发:
ch api conversation npc <玩家> <NPC> look
可与标签 FORCE_DISPLAY, NO_MOVE, NO_SKIP 三者结合达成无法跳过的对话。
该功能常用于纠正玩家的视线，以便后续 播放动画 或 展示部分效果。
同时此功能由指令触发时，玩家会强制目视NPC的方向，可用于制作带有藏匿元素的NPC。
FORCE_LOOK 强制目视 展示

FORCE_DISPLAY 强制对话
该标签用于在触发对话时，强制让玩家完成对话。
正常情况下，玩家在触发对话后进行大范围移动会打断对话进行。
使用该标签后，玩家无法再通过大范围移动打断对话进行。
FORCE_DISPLAY 强制对话 展示

LOOK_PLAYER 目视玩家
该标签用于在触发对话时，强制让NPC目视玩家进行对话。
当对话结束或中止后，NPC会强制恢复原视角。
LOOK_PLAYER 目视玩家 展示

NO_EFFECT 取消特效
该标签用于在触发对话时，强制取消某种特定特效或所有特效效果。
该标签分为一个父标签:
NO_EFFECT
及四个子标签:
NO_EFFECT:SLOW
NO_EFFECT:BLINDNESS
NO_EFFECT:SOUND
 NO_EFFECT:PARTICLE
四个子标签可分别独立使用。
为保持文件格式规范，同时减少后续维护难度，编者强烈建议在使用父标签时，不填写任何子标签。
注：此处的父子标签为编者提出，仅为用于减低讲解难度。
接下来我们将从四个子标签作为起点开始讲解。

配置文件结构拆解
接下来是一个简单的示例：
10_diamond_ore:
  meta:
    name: Break Diamonds
    type: L1
  task:
    0:
      objective: block break
      condition:
        material: diamond_ore
      goal:
        amount: 1
这便是一个基础的任务文件，让我们自上而下地对它进行分析。
任务标识符
首先填写的内容是 任务标识符，即 10_diamond_ore。其在 Chemdah 中对任务起区分作用，因此 任务标识符 具有唯一性，不可重复。
插件重载时会重新读取任务，当我们使用插件指令 /chemdah quest 时，Chemdah会自动将已读取成功的任务以 任务标识符 的形式为我们展示，以便于补全指令。
示例：

Meta
Meta 分为两个部分：name(名称)、type(类型)。
Name 名称
  meta:
    name: Break Diamonds
此处所填写的名称即为 Break Diamonds。
虽然名称的填写规则很宽松，但编者建议，在填写名称时自行协定一套命名规则。
Type 类型
  meta:
    type: L1
此处所填写的类型即为 L1，类型可用于任务组分类及 Chemdah 自带的UI菜单系统。
类型与名称一般，没有强制的命名规则，仅需按需求分类即可。

Task
  task:
    0:
      objective: block break
      condition:
        material: diamond_ore
      goal:
        amount: 1
task作为任务不可或缺的部分，在任务文件中起确定任务目标的作用。
任务条目
任务条目用于分隔多个任务目标，而此处的 0 即为 任务条目。
objective 任务类型
objective 用于选取任务类型，单个任务条目内仅可使用一个 objective。
多个任务条目可使用相同的 objective，用于以形成阶段性任务。
condition 任务条件
condition 在任务中是可有可无的，当其生效时，常用于限制特定条件。
各任务类型的 condition 各不相同，因此本文不再赘述 condition 的内容。
如有需求，请移步 任务示例 查看。
goal 任务目标
值得一提的是，大部分任务类型都支持在 goal 中进行 Kether 代理($)。

Addon
Addon 是任务的组件部分，常用于对任务的进行各项限制。
Addon 可用于任务条目及任务全局，但条目和全局的组件存在些许差异。

条目组件
条目组件内包含的独有内容仅可在条目内使用。
条目组件

depend 依赖
依赖，顾名思义是用于识别前置任务是否完成的。
当填写的前置任务尚未完成时，即使玩家完成了该条目的条件，依赖仍会控制条目无法完成。
party 队伍
队伍，用于为玩家开启组队任务，同时决定任务目标是否会同步给队伍中的所有玩家。
可配置只有在队伍中且成员数量达到要求时才可以继续任务。
该组件需要额外插件，详情请查看 Chemdah/module/party.yml 内的可用插件。
restart 重置
重置，用于在特定时刻重置任务数据。
当 Kether 动作当结果为 true 时重置任务数据。
stats 状态
状态，用于任务或任务条目的进度显示。
具体展现形式为 BossBar 血条。
timeout 超时
超时，用于对任务进行完成时间的限制。
当到达限制时间后，任务失败。
track 追踪
追踪，用于对任务目标进行导航或信息提示。
追踪包含多种目标导航方式。
optional 可选
可选，用于对任务条目的完成进行可选化操作。
全局组件

ui 模块作为 Chemdah 的任务视图部分，其可通过任务类型及 ui 任务页面 的进行联动，从而完成对特定任务的进度展示。
ui 模块从任务界面组出发，通过对 name(标题)、menu(布局)、item(物品)、include(包含选择器)、exclude(排除选择器) 的配置、编排，从而完成自定义任务视图的作用。
Chemdah 作为单纯的服务端界面，是无法实现
任务界面组
任务界面组作为识别 任务页面 的标识符，需在全局中保持唯一，即具有唯一性，不可重复。
建议使用任务类型或任务难易进行分类。
name 标题
name，即打开 任务页面 时，箱子菜单上展示的标题名。
此处提供了两个变量，即 {name} 和 {page}，分别对应 玩家名称 和 页码。

menu 布局

quest 任务
该处的 quest 任务，指的是即将显示在 任务页面 的槽位。
